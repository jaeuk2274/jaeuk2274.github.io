## 1. 시스템 소개

### 1-1. 인사시스템 개요

우리 회사의 통합 인사시스템은 다음 기능을 하나의 플랫폼에서 제공합니다.

- **인사관리**: 발령, 직급, 호봉, 인사이력 관리
- **근태관리**: 출퇴근, 연장/야간/휴일근로, 연차 관리
- **급여/상여 계산**: 월 급여, 분기 상여, 소급, 퇴직금 일괄 처리
- **모바일 연동**: 근태 신청, 급여명세서 조회, 증명서 발급

```
[사용자 접근 경로]
웹 브라우저  ──┐
모바일 앱    ──┼──→ Spring WAS ──→ Oracle DB (인사시스템스키마)
사내 인트라넷 ──┘                      └── 패키지: 급여계산엔진
                                           └── 패키지: 일별근태계산
                                           └── 패키지: 급여보조계산
```

### 1-2. 급상여 업무 프로세스

급여 계산은 단순한 수식 계산이 아닙니다.  
**근태 마감 → 급여 계산 → 상여 계산 → 소급 처리 → 마감** 의 순서로 진행됩니다.

```
[월별 급상여 처리 흐름]

① 근태 마감 (매월 15일 기준)
   일별근태계산.근태생성()
     └─ 직원 1명 × 1일: 연장/야간/휴일 시간 6종 산출
     └─ 대상: 3,000명 × 90일(3개월 소급) = 270,000건 

② 근태 집계 마감
   근태집계 (근태마감 테이블 집계)
     └─ 잔여연차, 연차수당, 연장누계 산출
     └─ 대상: 3,000명 × 3건 = 9,000건

③ 급여 계산 (급여구분코드='10')
   급여계산엔진.메인()
     └─ 주요 수당 45항목, 주요 공제 22항목 처리
     └─ 대상: 3,000명 × 301건 = 903,000건

④ 상여 계산 (분할하여 매달)
   급여계산엔진.메인() — 상여 모드
     └─ 급여와 거의 동일한 엔진 사용 (주요 수당 45항목, 주요 공제 22항목), 비율 등 특별 로직 일부
     └─ OT 일별 계산(일별통상시급 × 150회)은 상여 특성상 적거나 없음
     └─ 대상: 3,000명 × 약 150~200건 = 약 45~60만 건

> **1인당 301건 산출 근거**
>
> | 계산 단계 | 건수 | 비고 |
> |----------|------|------|
> | 통상임금 구성 합산 | 12건 | 기본급·직능기초급·직능가급·근속급 등 12개 항목 |
> | 통상시급 산출 | 1건 | 통상임금 합계 ÷ 소정근로시간 |
> | 임금피크율 적용 | 1건 | 통상임금 × 피크율 / 100 |
> | 일별통상시급() 호출 | 150건 | OT 발생일 최대 30일 × 수당 종류 5개 |
> | 수당 EXECUTE IMMEDIATE | 45건 | 수당항목정의 수식 동적 파싱·실행 |
> | 수당 임금피크 적용 | 30건 | 임금피크 대상 항목별 피크율 곱산 |
> | 수당 일할공제 | 35건 | 중도입퇴사·감봉 대상 항목 일할 계산 |
> | 공제 EXECUTE IMMEDIATE | 22건 | 공제항목정의 수식 동적 파싱·실행 |
> | 세금 계산 | 5건 | 소득세·지방소득세·건강·장기요양·고용보험 |
> | **합계** | **301건** | |

⑤ 소급 계산 (임단협 타결 시)
   급여계산엔진.소급계산()
     └─ 과거 N개월 급여 전체 재계산 후 차액 기록
     └─ 소급 발생 월 최대: 3,000명 × 90일(3개월치) = 270,000건 (급여 재계산과 동일 엔진)

⑥ 마감 처리
   급여마감처리.메인()
     └─ 계좌 배분, 연차수당 연결, 채권압류 잔액 UPDATE
```

**월 전체 계산 건수 (정상 월 기준)**

| 업무 | 대상 | 월 계산 건수 |
|------|------|------------|
| 근태 처리 | 3,000명 × 90일(3개월) | 270,000건 |
| 급여 계산 | 3,000명 × 301건 | 903,000건 |
| 상여 계산 | 3,000명 × 분기 | 약 50만 건 |
| 퇴직금 계산 | 월 10명 | 5,110건 |
| **합계** | | **약 170만 건/월** |

> **소급 계산** 실 수혜 대상자가 적어(소급수당: -2개월치 급상여 재계산/실 지급급여 비교),  
> **임단 소급**(매년 임금협상이 10월~11월 확정, -10개월 재계산)은 연 1회 발생하므로 위 합계에서 제외했습니다.

## 2. 이슈 발단

### 2-1. 측정 — 얼마나 걸리나

간단한 측정을 먼저 했습니다.

```sql
-- 급여 계산 1인당 소요시간 측정
DECLARE
  l_시작시간  TIMESTAMP;
  l_종료시간  TIMESTAMP;
  l_오류메시지 VARCHAR2(2000);
  l_반환코드   NUMBER;
BEGIN
  l_시작시간 := SYSTIMESTAMP;

  급여계산엔진.메인(
    오류메시지  => l_오류메시지,
    반환코드    => l_반환코드,
    p_급여구분코드 => '10',
    p_급여년월    => '202504',
    p_급여일      => TO_DATE('20250425','YYYYMMDD'),
    p_직원번호    => 1001   -- 테스트 직원 1명
  );

  l_종료시간 := SYSTIMESTAMP;
  DBMS_OUTPUT.PUT_LINE('1인 소요: ' ||
    TO_CHAR(EXTRACT(SECOND FROM (l_종료시간 - l_시작시간)), 'FM999.999') || '초');
END;
```

**결과: 직원 1명당 평균 3~4초**

```
전체 3,000명 직렬 처리 예상 시간:
  3,000명 × 3.6초 = 10,800초 ÷ 3,600 = 약 3시간

현실에서는 급여 계산 버튼 누르고 3시간 뒤에야 완료
```

### 2-2. 첫 번째 의심 — 병렬로 돌리면 안 되나?

처음 떠오른 아이디어는 단순했습니다.

> *"Java에서 멀티스레드로 여러 명 동시에 계산하면 빠르지 않을까?"*  

결론부터 말하면, **Java 레이어의 비동기화만으로는 근본 해결이 안 됩니다.**

그 이유를 이해하려면 급여 계산 엔진 내부를 봐야 합니다.

## 3. 문제 진단

### 3-1. 급여 계산 엔진 전체 구조

`급여계산엔진.메인()` 은 직원 1명의 급여를 처음부터 끝까지 처리하는 **모노리식 PL/SQL 프로시저**입니다.

```
급여계산엔진.메인(p_직원번호 => :직원ID)
  │
  ├─ 1. 사원마스터 로드 (사원마스터)
  ├─ 2. 근태집계 로드 (근태집계)
  ├─ 3. 임금피크 적용률 계산 (임금피크율계산)
  ├─ 4. 지급역 및 근무일수 산정
  │
  ├─ 5. 수당 계산 루프 (수당항목정의 45항목)
  │     └─ 항목별 EXECUTE IMMEDIATE 수식 실행
  │     └─ 수당상세저장() → 수당상세 INSERT
  │
  ├─ 6. 급여헤더 INSERT (급여헤더)
  │
  ├─ 7. 공제 계산 루프 (공제항목정의 22항목)
  │     └─ 항목별 EXECUTE IMMEDIATE 수식 실행
  │     └─ 공제상세저장() → 공제상세 INSERT
  │
  ├─ 8. 세금 계산 (세금계산엔진.메인)
  │     ├─ 소득세 간이세액 산출
  │     ├─ 지방소득세 = 소득세 × 10%
  │     └─ 장기요양보험 = 건강보험 × 요율
  │
  ├─ 9. 차인지급액 = 수당합계 - 공제합계
  │
  └─ 10. 급여마감처리.메인() — 마감처리
```

### 3-2. 직원 1명 계산 = 몇 건의 DB 연산인가

실제 코드를 분석한 결과, **직원 1명당 301건의 산술 연산**이 발생합니다.

| 계산 항목 | 1인당 건수 | 핵심 내용 |
|-----------|-----------|----------|
| 통상임금 구성 합산 | 12건 | 기본급+직능기초급+직능가급+근속급 등 |
| 통상시급 산출 | 1건 | 월통상임금 ÷ 소정근로시간 |
| 임금피크율 적용 | 1건 | 통상임금 × 피크율 / 100 |
| **일자별 통상시급** | **150건** | OT 발생일마다 `일별통상시급()` 호출 |
| **수당 EXECUTE IMMEDIATE** | **45건** | 수당항목정의 수식 동적 파싱·실행 |
| 수당 임금피크 적용 | 30건 | 임금피크플래그='Y' 항목 × 피크율 곱산 |
| 수당 일할공제 | 35건 | 중도입퇴사/감봉 일할 계산 |
| **공제 EXECUTE IMMEDIATE** | **22건** | 공제항목정의 수식 동적 파싱·실행 |
| 세금 계산 | 5건 | 소득세+지방세+장기요양 |
| **합계** | **301건** | |

**병목의 핵심은 세 가지:**

```
① EXECUTE IMMEDIATE × 67회 (수당45 + 공제22)
   → 수식 문자열을 매번 Oracle이 파싱·실행
   → Shared Pool 경합의 주범

② 일별통상시급() × 150회
   → OT 발생일마다 사원이력 테이블 조회
   → 1인당 최대 150번 같은 테이블 반복 접근

③ 급여배치처리: 3,000명을 1명씩 직렬 처리
   → CPU 1코어만 사용, 나머지 15개 유휴
```

## 4. 문제점 상세 (1) — PL/SQL 내부 반복 계산

### 4-1. EXECUTE IMMEDIATE × 67회 — "수식을 매번 즉석 요리"

#### 배경: 왜 수식을 테이블에 저장했나?

급여 수당 항목(식대, 직책수당, 연장수당 등)은 회사 규정 변경, 단체협약, 임금체계 개편 등으로 **계산 공식이 자주 바뀝니다.**
그래서 수식 자체를 코드에 하드코딩하지 않고, DB 테이블(수당항목정의, 공제항목정의)에 문자열로 저장한 뒤 런타임에 실행하는 구조로 되어 있었습니다.

```sql
-- 수당항목정의 테이블 (수당 수식 관리)
-- 수당코드  상세수식
-- -------  --------------------------------------------------
-- A01      기본급계산() * NVL(:임금피크율, 100) / 100
-- A11      일할수당계산() * :근무일수 / :총일수
-- A13      직능기초급계산() * NVL(:임금피크율, 100) / 100
-- ...      (수당 45항목, 공제 22항목)
```

#### 실제 실행 코드

```sql
FUNCTION 수당항목계산(p_수당코드 VARCHAR2) RETURN NUMBER IS
  l_변환수식    VARCHAR2(2000);
  l_동적SQL     LONG;
  l_금액        NUMBER;
BEGIN
  IF (g_숨김수식 IS NOT NULL) THEN
    수식변환(사원정보.직원번호, g_숨김수식, l_변환수식);

    -- ★ 매 항목마다 새로운 SQL 문자열 조립 → Oracle이 매번 파싱
    EXECUTE IMMEDIATE 'SELECT ' || l_변환수식 || ' FROM DUAL'
      INTO l_금액;

  ELSIF (g_상세수식 IS NOT NULL) THEN
    -- 수당계산팩_{기관코드}_{회사코드}_{수당코드}() 형태로 동적 조립
    l_동적SQL := 'BEGIN ';
    l_동적SQL := l_동적SQL || ' 수당계산팩_' || g_기관코드
                           || '_' || g_회사코드 || '_' || p_수당코드 || '( ';
    l_동적SQL := l_동적SQL || '    :p_계산유형, :p_기관코드, :p_회사코드, ';
    l_동적SQL := l_동적SQL || '    :p_급여구분코드, :p_급여년월, :p_급여일, ';
    l_동적SQL := l_동적SQL || '    :p_직원번호, :p_이력번호, ';
    -- ... (바인드 변수 60개 이상 계속) ...
    l_동적SQL := l_동적SQL || ' ); END;';

    -- ★ 런타임에 조립된 SQL 실행 — 매 항목마다 새로운 파싱 발생
    EXECUTE IMMEDIATE l_동적SQL
      USING IN g_계산모듈, IN g_기관코드, ...
            OUT l_금액;
  END IF;

  RETURN l_금액;
END 수당항목계산;
```

#### 성능 문제

```
직원 3,000명 × 67회 EXECUTE IMMEDIATE = 201,000번 동적 파싱
→ Oracle Shared Pool 경합
→ Library Cache Latch 대기 발생
→ 배치 실행 중 v$DB대기이벤트에서 라이브러리캐시잠금 대기 급증
```

### 4-2. 일별통상시급() × 150회 — "그날의 시급을 매번 DB에서 꺼내온다"

#### 왜 일자별로 통상시급을 다시 계산해야 하나?

```
예시: 홍길동 직원 (5월 15일 승진)
  5월 1일  연장근무 → 승진 전 호봉 기준 통상시급 필요
  5월 16일 연장근무 → 승진 후 호봉 기준 통상시급 필요
  → 날짜별로 사원이력 테이블 참조 필수
```

#### 실제 호출 코드

```sql
-- 급여계산엔진.메인() 내부 야간수당 계산
SELECT 야간수당금액 * 0.5 * l_야간시간수 INTO l_야간수당금액
  FROM (
    SELECT d.근무일자,
           -- ★ 날짜가 바뀔 때마다 함수 호출 → 사원이력 조회 발생
           급여보조계산.일별통상시급(
               d.직원번호, g_이력번호, d.근무일자
           ) 야간수당금액
      FROM 근태상세 d
     WHERE d.근무일자 BETWEEN g_급여시작일 AND g_급여종료일
       AND 직원번호 = 사원정보.직원번호 AND NVL(d.야간시간수, 0) > 0
  );
-- 연장수당, 휴일수당, 휴일초과, 휴일야간 각각에도 동일 패턴
-- → OT 발생 날짜 30일 × 수당 종류 5개 = 150회 호출
```

#### 일별통상시급() 내부


```sql
FUNCTION 일별통상시급(p_직원번호     사원마스터.직원번호%TYPE,
                      p_이력번호     사원이력.이력번호%TYPE,
                      p_기준일       DATE)
RETURN NUMBER IS
  l_소정근로시간 NUMBER DEFAULT 공통코드.공통값('REGULAR_WORK_TIME');
  -- 소정근로시간: 월 209시간 (일반직 기준)
BEGIN
  -- ① 매 호출마다 사원이력 테이블 조회 ← DB I/O 발생
  SELECT * INTO 사원정보
    FROM 사원이력 h
   WHERE 직원번호 = p_직원번호 AND 이력번호 = p_이력번호;

  -- ② p_기준일로 이 날짜가 속하는 급여 기간(g_급여년월, g_급여시작일/종료일) 설정
  IF to_char(p_기준일, 'DD') <= '15' THEN
    g_급여년월   := to_char(add_months(p_기준일, -1), 'RRRRMM');  -- 예) 5/1 → 202504
    g_급여시작일 := 전월 16일;
    g_급여종료일 := 당월 15일;
  ELSE
    g_급여년월   := to_char(p_기준일, 'RRRRMM');                  -- 예) 5/20 → 202505
    g_급여시작일 := 당월 16일;
    g_급여종료일 := 익월 15일;
  END IF;
  -- g_급여년월은 이후 임금피크율 조회, 호봉 조회 등의 기준으로 사용됨

  -- ③ 임금피크율 조회 (g_급여종료일 기준)
  g_임금피크율 := 급여계산엔진.임금피크율조회(p_직원번호, g_급여종료일, g_급여년월, '1C');

  -- ④ 통상임금 구성 12개 항목 각각 계산 (사원정보 전역변수에서 읽음)
  l_기본급금액       := 기본급계산();
  l_직능기초급금액   := 직능기초급계산();
  l_직능가급금액     := 직능가급계산();
  l_근속급금액       := 근속급계산();
  l_임금보전급금액   := 임금보전급계산();
  l_중식비금액       := 중식비계산();
  l_교대수당금액     := 교대수당계산();
  l_직책수당금액     := 직책수당계산();
  l_기초수당금액     := 기초수당계산();
  l_직무환경수당금액 := 직무환경수당계산();
  l_기타수당금액     := 기타수당계산();
  l_보전수당금액     := 보전수당계산();

  l_금액 := nvl(l_기본급금액,0)       + nvl(l_직능기초급금액,0)
           + nvl(l_직능가급금액,0)     + nvl(l_근속급금액,0)
           + nvl(l_임금보전급금액,0)   + nvl(l_중식비금액,0)
           + nvl(l_교대수당금액,0)     + nvl(l_직책수당금액,0)
           + nvl(l_기초수당금액,0)     + nvl(l_직무환경수당금액,0)
           + nvl(l_기타수당금액,0)     + nvl(l_보전수당금액,0);

  -- ⑤ 통상임금 ÷ 소정근로시간 = 통상시급 반환
  --    (함수명은 일별이지만 실제 반환값은 시급)
  l_금액 := l_금액 / l_소정근로시간;  -- 예: 300만 / 209h = 14,354원/h
  RETURN l_금액;
  -- 호출부에서 이 시급에 시간을 곱해 OT 수당 계산
  -- 예: 야간수당 = 시급 × 야간시간 × 0.5
END;
```

## 6. 개선안 (1) — PL/SQL Associative Array 캐시

### 6-1. 왜 RESULT_CACHE를 쓰지 않았나

Oracle 11g 이상에서 `RESULT_CACHE` 키워드를 함수에 붙이면 자동으로 결과를 캐싱할 수 있습니다.  
하지만 `일별통상시급()` 에는 적용할 수 없었습니다. 이유를 이해하려면 이 함수가 **두 가지 역할을 동시에 수행**한다는 것을 먼저 알아야 합니다.

#### RESULT_CACHE의 전제 — "순수 함수"여야 한다

```sql
-- ✅ RESULT_CACHE에 적합한 순수 함수 예시
FUNCTION 세금계산(p_금액 NUMBER) RETURN NUMBER
RESULT_CACHE IS
BEGIN
  RETURN p_금액 * 0.033;  -- 입력값만으로 결과가 결정됨
END;
-- 호출1: 세금계산(1000000) → 실행 → 33,000 SGA 캐시 저장
-- 호출2: 세금계산(1000000) → 캐시 히트 → 함수 본문 통째로 스킵 → 33,000 반환
-- ← 안전. 이 함수는 아무 전역변수도 "설정"하지 않으므로
```

#### 일별통상시급()은 "계산"과 "설정"을 동시에 한다

```sql
-- 급여보조계산 패키지 전역변수
PACKAGE BODY 급여보조계산 IS
  사원정보  사원이력%ROWTYPE;  -- ★ 패키지 전역변수 (모든 하위 함수가 공유)

  FUNCTION 일별통상시급(p_직원번호 NUMBER, p_이력번호 NUMBER, p_기준일 DATE)
  RETURN NUMBER IS
  BEGIN
    -- [역할1] 사이드이펙트: 전역변수 사원정보에 직원 이력 "설정"
    SELECT * INTO 사원정보              -- ← 패키지 전역변수에 직접 저장
      FROM 사원이력
     WHERE 직원번호  = p_직원번호
       AND 이력번호  = p_이력번호;

    -- [역할2] 설정된 사원정보를 읽어서 통상시급 "계산" 후 반환
    RETURN (기본급계산()       -- ← 사원정보.기본급 읽음
          + 직능기초급계산()   -- ← 사원정보.직능기초급 읽음
          + 근속급계산()       -- ← 사원정보.근속연수 읽음
          + ...) / l_소정근로시간;
  END;

  -- 하위 함수들도 같은 전역변수 사원정보를 직접 읽음
  FUNCTION 기본급계산 RETURN NUMBER IS
  BEGIN
    RETURN 사원정보.기본급;  -- ← 전역변수 직접 참조
  END;
END;
```

#### 문제 시나리오 — 5월 15일 승진한 직원 (이력번호 2개)

홍길동이 5월 15일에 승진하면 이력번호가 두 개 생깁니다.

```
이력번호 5: 승진 전 (기본급 300만, 직급 과장)  → 5/1 ~ 5/14 OT에 사용
이력번호 6: 승진 후 (기본급 350만, 직급 차장)  → 5/15 ~ 5/31 OT에 사용
```

급여계산엔진.메인() 내부에서는 수당 종류별로(연장, 야간, 휴일...) 각 날짜를 반복 호출합니다.

```sql
-- ❌ RESULT_CACHE 적용 시 실제 호출 흐름

-- [연장수당 계산 루프]
일별통상시급(홍길동, 이력번호=5, 5/1)   -- CACHE MISS → 실행
  → 사원정보 = {기본급: 300만, 직급: 과장}  로드됨
  → 반환값 100,000원 → SGA 캐시 저장

일별통상시급(홍길동, 이력번호=6, 5/20)  -- CACHE MISS → 실행
  → 사원정보 = {기본급: 350만, 직급: 차장}  로드됨 (전역변수 덮어씀)
  → 반환값 116,000원 → SGA 캐시 저장

-- [야간수당 계산 루프 — 같은 날짜를 다시 호출]
일별통상시급(홍길동, 이력번호=5, 5/1)   -- ★ CACHE HIT → 함수 본문 통째로 스킵!
  → 반환값: 100,000원 (캐시에서)           -- 값은 맞음
  → 사원정보: {기본급: 350만, 직급: 차장} 그대로  -- 이력6 상태로 오염! ❌

-- 이 시점에 기본급계산() 등 하위 함수를 직접 호출하면?
SELECT 기본급계산() INTO l_기본급 FROM DUAL;
  → 사원정보.기본급 읽음 → 350만 반환  -- 5/1(승진 전)인데 승진 후 기본급 사용 ❌
```

```
핵심 정리:
  RESULT_CACHE의 전제: "입력값이 같으면 함수 본문을 실행할 필요 없다"
  일별통상시급의 현실:
    반환값    → 입력값(직원번호, 이력번호, 기준일)으로 결정 ← RESULT_CACHE 입장에선 OK
    사원정보  → 입력값으로 결정되지만 "전역변수"에 저장됨
                                      ↑
              RESULT_CACHE는 이걸 복원해주지 않음
              캐시 히트 후 사원정보는 직전 다른 호출이 남긴 값

  결과: 반환값은 정확하지만, 전역변수 사원정보가 오염된 채로 후속 계산 진행
```

### 6-2. 해결책 — 패키지 레벨 Associative Array 캐시

함수 내부에서 직접 캐싱 로직을 구현합니다.  
Oracle PL/SQL의 `Associative Array (INDEX BY)` 를 패키지 전역변수로 선언하면  
**같은 세션 내에서 호출 간에 결과를 공유**할 수 있습니다.

단순히 `통상시급` 숫자만 캐시하면 `RESULT_CACHE`와 같은 문제가 생깁니다.  
그래서 캐시에는 **반환값뿐 아니라 함수 본문에서 설정하던 계산 컨텍스트**도 함께 저장했습니다.  
캐시 히트 시에는 이 컨텍스트를 패키지 전역변수에 먼저 복원한 뒤 값을 반환합니다.

#### 구현 코드 — 급여보조계산 패키지 수정

```sql
-- ① PACKAGE BODY 상단 전역변수 선언부에 캐시 타입 추가
PACKAGE BODY 급여보조계산 IS

  -- 기존 전역변수들...
  g_처리단계     VARCHAR2(300);
  사원정보       사원이력%ROWTYPE;
  g_임금피크율   NUMBER DEFAULT 100;
  -- ...

  -- ★ 신규 추가: 캐시 타입 및 변수 선언
  -- 키: '직원번호_이력번호_YYYYMMDD_파라미터' 조합 문자열
  -- 값: 일별통상시급 반환값 + 계산 컨텍스트
  TYPE t_금액캐시형     IS TABLE OF NUMBER          INDEX BY VARCHAR2(80);
  TYPE t_사원정보캐시형 IS TABLE OF 사원이력%ROWTYPE INDEX BY VARCHAR2(80);
  TYPE t_문자캐시형     IS TABLE OF VARCHAR2(20)    INDEX BY VARCHAR2(80);
  TYPE t_일자캐시형     IS TABLE OF DATE            INDEX BY VARCHAR2(80);

  g_통상시급캐시   t_금액캐시형;
  g_사원정보캐시   t_사원정보캐시형;
  g_급여년월캐시   t_문자캐시형;
  g_급여시작일캐시 t_일자캐시형;
  g_급여종료일캐시 t_일자캐시형;
  g_임금피크율캐시 t_금액캐시형;

  -- ② 일별통상시급 함수 수정
  FUNCTION 일별통상시급(p_직원번호    사원마스터.직원번호%TYPE,
                        p_이력번호    사원이력.이력번호%TYPE,
                        p_기준일      DATE,
                        p_파라미터    VARCHAR2 DEFAULT NULL)
  RETURN NUMBER IS

    l_캐시키    VARCHAR2(80);
    l_금액      NUMBER;

  BEGIN
    -- 캐시 키 생성: 같은 직원 + 같은 이력번호 + 같은 날짜 + 같은 파라미터
    l_캐시키 := TO_CHAR(p_직원번호)
             || '_' || TO_CHAR(p_이력번호)
             || '_' || TO_CHAR(p_기준일, 'YYYYMMDD')
             || '_' || NVL(p_파라미터, '실지급');

    -- ★ 캐시 히트: DB 조회 없이 계산 컨텍스트 복원 후 반환
    IF g_통상시급캐시.EXISTS(l_캐시키) THEN
      사원정보     := g_사원정보캐시(l_캐시키);
      g_급여년월   := g_급여년월캐시(l_캐시키);
      g_급여시작일 := g_급여시작일캐시(l_캐시키);
      g_급여종료일 := g_급여종료일캐시(l_캐시키);
      g_임금피크율 := g_임금피크율캐시(l_캐시키);

      RETURN g_통상시급캐시(l_캐시키);
    END IF;

    -- ★ 캐시 미스: 기존 로직 그대로 실행
    -- ① 사원이력 조회
    SELECT * INTO 사원정보
      FROM 사원이력 h
     WHERE 직원번호  = p_직원번호
       AND 이력번호  = p_이력번호;

    -- ② 날짜 기준 기간 계산
    IF TO_CHAR(p_기준일, 'DD') <= '15' THEN
      g_급여년월   := TO_CHAR(ADD_MONTHS(p_기준일,-1), 'RRRRMM');
      g_급여시작일 := TO_DATE(TO_CHAR(ADD_MONTHS(p_기준일,-1),'RRRRMM')||'16','RRRRMMDD');
      g_급여종료일 := TO_DATE(TO_CHAR(p_기준일,'RRRRMM')||'15','RRRRMMDD');
    ELSE
      g_급여년월   := TO_CHAR(p_기준일, 'RRRRMM');
      g_급여시작일 := TO_DATE(TO_CHAR(p_기준일,'RRRRMM')||'16','RRRRMMDD');
      g_급여종료일 := TO_DATE(TO_CHAR(ADD_MONTHS(p_기준일,1),'RRRRMM')||'15','RRRRMMDD');
    END IF;

    -- ③ 급여 기간 기준 임금피크율 계산
    g_임금피크율 := 급여계산엔진.임금피크율조회(
                      p_직원번호, g_급여종료일, g_급여년월, '1C');

    -- ④ 통상임금 구성 12개 항목 계산 (기존 로직 그대로)
    l_금액 := NVL(기본급계산(),0)       + NVL(직능기초급계산(),0) + NVL(직능가급계산(),0)
            + NVL(근속급계산(),0)        + NVL(임금보전급계산(),0) + NVL(중식비계산(),0)
            + NVL(교대수당계산(),0)      + NVL(직책수당계산(),0) + NVL(기초수당계산(),0)
            + NVL(직무환경수당계산(),0)  + NVL(기타수당계산(),0) + NVL(보전수당계산(),0);

    -- ★ 계산 결과와 컨텍스트를 캐시에 저장 (이후 동일 키 요청 시 재사용)
    g_통상시급캐시(l_캐시키)   := l_금액 / l_소정근로시간;
    g_사원정보캐시(l_캐시키)   := 사원정보;
    g_급여년월캐시(l_캐시키)   := g_급여년월;
    g_급여시작일캐시(l_캐시키) := g_급여시작일;
    g_급여종료일캐시(l_캐시키) := g_급여종료일;
    g_임금피크율캐시(l_캐시키) := g_임금피크율;

    RETURN g_통상시급캐시(l_캐시키);

  EXCEPTION
    WHEN OTHERS THEN
      RETURN 0;
  END 일별통상시급;

  -- ③ 캐시 초기화 프로시저 추가
  --    급여계산엔진.메인() 에서 직원 1명 처리 완료 후 호출
  PROCEDURE 일별캐시초기화 IS
  BEGIN
    g_통상시급캐시.DELETE;
    g_사원정보캐시.DELETE;
    g_급여년월캐시.DELETE;
    g_급여시작일캐시.DELETE;
    g_급여종료일캐시.DELETE;
    g_임금피크율캐시.DELETE;
  END 일별캐시초기화;

END 급여보조계산;
```

## 7. 개선안 (2) — 동적 병렬처리와 커넥션 제어

### 7-1. 설계 방향 — CPU 조회 → 그룹 분할 → 일괄 병렬 실행

배치 시작 시점에 **DB CPU를 한 번 조회**해서 스레드 수를 결정하고,  
전체 직원을 그 수만큼 그룹으로 나눠 **ExecutorService로 동시에 실행**합니다.

```
① DB CPU 조회 (시작 시 1회)
   CPU 35% → 스레드 12개
   CPU 60% → 스레드  6개
   CPU 80% → 스레드  1개

② 전체 직원 3,000명 → N개 그룹 분할
   스레드 12개 → 그룹당 약 250명

③ N개 그룹 동시 실행 (ExecutorService+Future)
   [그룹1-250명] [그룹2-250명] ... [그룹12-250명]
   각 그룹 = 독립 DB 커넥션 = 독립 Oracle 세션

④ 전체 완료 대기 후 결과 취합
```

> 실행 중 CPU가 올라가도 스레드 수는 고정입니다.  
> 시작 시점의 CPU 기준으로 판단하므로, 업무시간 후 또는 근태마감 직후 여유 있을 때 돌리는 것이 좋습니다.

### 7-2. 파일 구조

```
src/main/java/com/company/hr/
├── batch/
│   └── 급여배치서비스.java   ← CPU 조회 + 그룹 분할 + 병렬 실행 + 결과 취합
└── domain/
    └── 급여대상VO.java       ← VO

WEB-INF/spring/
└── appContext-ds1.xml        ← DataSource XML 설정
```

> 스레드풀은 Spring Bean이 아닌 배치 실행 시점에 동적으로 생성하고,  
> 완료 후 즉시 반납합니다 (`executor.shutdown()`).  
> 급여는 근태마감 후 담당자가 화면에서 직접 실행합니다.

### 7-3. 급여배치서비스 — CPU 조회 → 스레드풀 생성 → 병렬 실행

```java
// batch/급여배치서비스.java
@Service
public class 급여배치서비스 {

    private static final Logger log = LoggerFactory.getLogger(급여배치서비스.class);

    @Autowired private JdbcTemplate jdbcTemplate;
    @Autowired private DataSource   dataSource;


    /**
     * ① DB CPU 조회 → 스레드 수 결정
     * ② 전체 직원 → N개 그룹 분할
     * ③ ExecutorService(N스레드) 생성 → 각 그룹 Callable로 제출
     * ④ Future.get()으로 전체 완료 대기 → 결과 취합
     */
    public void 급여배치실행(String 급여년월, String 급여구분코드,
                            Date 급여일) throws Exception {

        // ① CPU 조회 → 스레드 수 결정
        int 스레드수 = 스레드수결정();
        log.info("[배치] DB CPU 기반 스레드 수: {}개", 스레드수);

        // ② 대상 직원 조회
        List<급여대상VO> 대상목록 = 대상직원조회(급여년월, 급여구분코드);
        log.info("[배치] 대상: {}명 → {}개 그룹 분배", 대상목록.size(), 스레드수);

        // ③ N개 그룹 분할
        List<List<급여대상VO>> 그룹목록 = 그룹분할(대상목록, 스레드수);

        // ④ 스레드풀 생성 — CPU 기반 스레드 수만큼
        ExecutorService executor = Executors.newFixedThreadPool(스레드수);
        List<Future<List<급여대상VO>>> futures =
                new ArrayList<Future<List<급여대상VO>>>();
        try {
            // ⑤ 각 그룹을 Callable로 제출 → 즉시 병렬 실행 시작
            for (int i = 0; i < 그룹목록.size(); i++) {
                final List<급여대상VO> 그룹    = 그룹목록.get(i);
                final int              그룹번호 = i + 1;
                futures.add(executor.submit(new Callable<List<급여대상VO>>() {
                    public List<급여대상VO> call() throws Exception {
                        return 그룹처리(그룹, 그룹번호);
                    }
                }));
            }

            // ⑥ 전체 완료 대기 + 결과 취합
            int 성공수 = 0, 실패수 = 0;
            for (Future<List<급여대상VO>> future : futures) {
                for (급여대상VO 결과 : future.get()) { // 완료될 때까지 블로킹
                    if (결과.isSuccess()) 성공수++;
                    else                  실패수++;
                }
            }
            log.info("[배치] 완료 — 성공: {}명, 실패: {}명", 성공수, 실패수);

        } finally {
            executor.shutdown(); // 스레드풀 반납
        }
    }

    /** DB CPU % → 스레드 수 결정 */
    private int 스레드수결정() {
        // GRANT SELECT ON v_$sysmetric TO CMEHR; 필요
        try {
            Double cpu = jdbcTemplate.queryForObject(
                "SELECT ROUND(value,1) FROM v$sysmetric" +
                " WHERE metric_name='Host CPU Utilization (%)'" +
                "   AND intsize_csec=6000 AND ROWNUM=1",
                Double.class);
            if (cpu == null) cpu = 50.0;
            log.info("[배치] DB CPU: {}%", cpu);

            if      (cpu < 40.0) return 12;
            else if (cpu < 55.0) return  9;
            else if (cpu < 65.0) return  6;
            else if (cpu < 75.0) return  3;
            else                 return  1;
        } catch (Exception e) {
            log.warn("[배치] CPU 조회 실패 → 기본값 6개: {}", e.getMessage());
            return 6;
        }
    }    
}
```

### 7-4. 그룹 처리 — 독립 커넥션으로 급여계산엔진 호출

> **MyBatis XML 매퍼를 쓰지 않는 이유**  
> MyBatis Mapper를 `@Transactional` 없이 호출하면 SQL마다 커넥션을 획득·반납하므로 Oracle 세션이 바뀔 수 있고,  
> `@Transactional`을 붙이면 그룹 내 직원 전체가 하나의 트랜잭션이 되어 **1명 실패 시 전체 롤백** 문제가 생깁니다.  
> `dataSource.getConnection()`으로 커넥션을 독점하면 Oracle 세션이 유지되고, **직원 1명 단위 커밋/롤백**이 가능합니다.

```java
private List<급여대상VO> 그룹처리(List<급여대상VO> 그룹, int 그룹번호) {
    Connection 연결 = null;
    List<급여대상VO> 결과목록 = new ArrayList<급여대상VO>(그룹.size());
    try {
        연결 = dataSource.getConnection(); // 커넥션풀에서 1개 독점
        연결.setAutoCommit(false);
        for (급여대상VO 대상 : 그룹) {
            급여대상VO 결과 = 급여계산호출(연결, 대상);
            결과목록.add(결과);
            if (결과.isSuccess()) 연결.commit();
            else                  연결.rollback();
        }
    } catch (Exception e) {
        log.error("[그룹-{}] 예외: {}", 그룹번호, e.getMessage(), e);
    } finally {
        if (연결 != null) try { 연결.close(); } catch (SQLException ex) {}
    }
    return 결과목록;
}

private 급여대상VO 급여계산호출(Connection 연결, 급여대상VO 대상) {
    // BEGIN 급여계산엔진.메인(오류메시지=>:1, 반환코드=>:2, ... p_직원번호=>:7, ...); END;
    try (CallableStatement cs = 연결.prepareCall("BEGIN 급여계산엔진.메인(...); END;")) {
        cs.registerOutParameter(1, Types.VARCHAR); // 오류메시지 OUT
        cs.registerOutParameter(2, Types.NUMERIC); // 반환코드 OUT
        // IN 파라미터 바인딩 (급여년월, 급여구분코드, 직원번호 등)
        cs.execute();
        대상.setSuccess(cs.getInt(2) != 2);
        대상.setErrorMessage(cs.getInt(2) == 2 ? cs.getString(1) : null);
    } catch (SQLException e) {
        대상.setSuccess(false);
        대상.setErrorMessage(e.getMessage());
    }
    return 대상;
}
```

| `사원이력` DB I/O | 450,000회/월 | ~6,000회/월 | **98% 감소** |
| `일별통상시급()` 실행시간 | 함수 150회 × N ms | 함수 2~3회 × N ms | **~98% 단축** |
| 직원 1인당 처리시간 | 약 3.6초 | 약 2.2초 | **40% 단축** |
| 3,000명 직렬 기준 | 약 3시간 | 약 1.8시간 | 1.2시간 단축 |

#### 동적 병렬처리 (개선안 2) 단독 적용

아래 수치는 DB 부하가 낮고 시작 시점 CPU 기준으로 12스레드가 선택된 구간의 실측입니다.

| 항목 | 개선 전 | 개선 후 | 변화 |
|------|---------|---------|------|
| 동시 처리 세션 수 | 1개 | CPU 기준 1~12개 | **최대 12배** |
| 3,000명 처리시간 | 약 3시간 | 약 15분 | **~12배 단축** |

#### 1안 + 2안 동시 적용

```
처리시간:
  개선 전:  약 3시간
  1안만:    약 1.8시간
  2안만:    약 15분  (12스레드 실행 구간 기준)
  1안+2안:  약 10분  (12스레드 실행 구간 기준) ← 목표 달성

3,000명 × 301건 = 903,000건 처리를 10분 안에 완료
```

### 8-2. 배운 점

#### "Java 비동기화 = DB 성능 개선" 이 아니다

처음에는 단순히 병렬 처리만 적용하면 빨라질 거라고 생각했습니다.
하지만 **전체 처리 시간의 95% 이상이 DB 내부 PL/SQL 실행 시간**이었습니다.

```
Java 비동기/병렬화가 해결하는 것: 대기 시간 중복 제거, 스레드 낭비 방지
PL/SQL이 해결해야 하는 것: 실제 계산 시간 단축

→ 두 레이어 각각의 문제를 구분해서 접근해야 한다
```

#### 전역변수는 세션 단위 — Oracle의 특성을 이해하고 병렬화

패키지 전역변수가 세션 단위로 격리된다는 Oracle의 특성을 파악하고 나서야  
커넥션 분리 기반의 병렬화가 가능해졌습니다.

```
"이 코드를 병렬로 돌릴 수 있나?"
→ 먼저 상태(전역변수)의 격리 단위를 파악하라
→ Oracle 패키지 변수는 세션 단위 격리 → 커넥션 분리로 해결
```

#### 캐시는 만능이 아니다 — 사이드이펙트를 파악하라

`RESULT_CACHE` 한 줄이면 끝날 것 같았지만, 함수 내부의 전역변수 사이드이펙트 때문에  
수동 캐시 구현으로 방향을 바꿔야 했습니다.

```
캐시 적용 전 확인사항:
  1. 함수가 동일 입력에 동일 출력을 반환하는가? (순수 함수인가)
  2. 함수 내부에 전역 상태 변경(사이드이펙트)이 있는가?
  3. 캐시 히트 시 건너뛰어지는 코드가 없어야 할 부작용이 있는가?
```

#### 배치와 OLTP는 분리해서 생각해야 한다

같은 시스템에서 배치와 실시간 서비스가 공존할 때는  
**DB 레벨의 리소스 분리** 또는 **애플리케이션 레벨의 커넥션 제어**가 반드시 필요합니다.

```
선택지:
  A. Oracle Resource Manager — DB 레벨 CPU 할당 제어 (근본적)
  B. ExecutorService (CPU%-기반 고정 스레드풀) — App 레벨 병렬 수 제어 (현실적)
  C. 배치 전용 계정/스키마 분리 — 명확하지만 운영 복잡도 증가

우리 선택: B (기존 인프라 변경 없이 적용 가능, 빠른 배포)
```

## 마치며

이 글은 실제 운영 중인 인사시스템에서 경험한 내용을 정리한 것입니다.

급여 시스템은 정확성이 최우선이기 때문에, 성능 개선을 하면서도 **계산 결과의 정합성을 검증하는 것**이 가장 중요했습니다. 개선 적용 후에는 기존 결과와 비교 검증 쿼리를 반드시 돌렸고, 차이가 0원인 것을 확인한 후에야 운영에 반영했습니다.

```sql
-- 개선 전후 급여 결과 검증 쿼리
SELECT a.직원번호,
       a.수당합계 AS 개선전_수당합계,
       b.수당합계 AS 개선후_수당합계,
       a.수당합계 - b.수당합계 AS 차이금액
  FROM 급여헤더 a
  JOIN 급여헤더신규 b ON a.직원번호 = b.직원번호 AND a.급여년월 = b.급여년월
 WHERE a.급여년월 = '202504'
   AND ABS(a.수당합계 - b.수당합계) > 0  -- 차이 있는 직원만 조회
 ORDER BY ABS(a.수당합계 - b.수당합계) DESC;

-- 결과: 0 rows selected (전 직원 결과 동일 확인)
```

성능 문제는 항상 측정부터 시작해야 한다는 것,  
그리고 레이어별로 문제를 분리해서 접근해야 한다는 것을 다시 한번 확인한 경험이었습니다.
